## I. Introduction

The integration of Large Language Models (LLMs) into software development pipelines has fundamentally altered how code is produced. Tools such as GitHub Copilot, Cursor, and Claude Code now generate substantial portions of production software: Copilot alone contributes to an average of 46% of developers' code across all programming languages [1]. As agentic coding systems become more capable and autonomous, the human developer's role increasingly shifts from code authorship to code review — from producing to verifying.

This shift creates a new and underexplored risk. Developers reviewing AI-generated code rely, consciously or unconsciously, on readability as a proxy for quality. Readable code — code with descriptive identifiers, consistent naming conventions, appropriate length, and clear structure — signals to the reviewer that the code is well-reasoned and correct. Yet this signal may be fundamentally unreliable when applied to LLM-generated code. LLMs are trained to produce output that is linguistically fluent and stylistically polished [2], an objective that is largely independent of semantic correctness. A model that generates a logically flawed implementation does not generate ugly code — it generates clean, well-named, readable-looking code that fails silently.

The empirical evidence supports this concern. A recent large-scale study by Ye et al. [3] analyzed 5,869 code samples generated by five frontier LLMs and found that LLM-generated code achieves statistically higher readability scores than human-written code (p < 0.001, r = 0.398). The same study identifies what it terms a "novel critical correctness-readability gap," noting that models generating "functionally correct but cryptic code ultimately impose greater cognitive burdens on developers in code maintenance" and calling for future research to address it [3]. Concurrently, a survey of 72 studies on bugs in LLM-generated code found that semantic errors — faults that render code functionally incorrect while appearing syntactically sound — constitute more than 60% of observed failures [4]. The convergence of these two findings raises an urgent question: if LLM-generated code is both more readable and more semantically error-prone than human-written code, does high readability mislead developers into misplacing their trust?

This question is further sharpened by what we term the *verification paradox*. According to the Stack Overflow Developer Survey 2025, 96% of developers report that they do not fully trust AI-generated code without manual verification. Yet only 48% report that they always verify AI-generated code before committing it [5]. This gap between stated distrust and actual verification behavior suggests that some factor reduces the perceived urgency of verification — and readable-looking code is a plausible candidate. Microsoft Research has separately found that developers miss 40% more bugs when reviewing AI-generated code compared to human-written code [5], consistent with a readability-induced reduction in critical scrutiny.

Despite this context, no prior study has empirically measured the *correlation* between multi-dimensional cognitive readability scores and functional correctness within LLM-generated code. Existing work either measures readability without correctness labels [3], or measures correctness without readability analysis [6]. The relationship between these two dimensions — and specifically the risk that readable code signals correctness it does not possess — remains unquantified.

This paper addresses that gap. We apply the IRAF-XADL cognitive readability framework [7] — a published identifier-level readability assessment system achieving 98%+ classification accuracy using CodeBERT embeddings and Self-Attention BiLSTM — to a large sample of LLM-generated Python code with ground-truth correctness labels derived from rigorous test suites. We introduce the **Deceptive Readability Index (DRI)**, a novel composite metric that quantifies the degree to which a code sample is readable-yet-incorrect. We then conduct a systematic empirical investigation of whether readability predicts correctness, how the readability-correctness gap varies across LLM architectures, and which specific cognitive readability parameters are most strongly associated with deceptive readability.

The contributions of this paper are:

1. **The first empirical study** measuring the correlation between multi-dimensional cognitive readability scores and functional correctness within LLM-generated code on standardized benchmarks.

2. **The Deceptive Readability Index (DRI)**, a novel composite metric that identifies code samples posing the highest false-trust risk to human reviewers.

3. **Per-parameter attribution of deceptive readability**: identification of which of the ten IRAF-XADL cognitive parameters (MC, NC, OL, DR, PR, LF, CC, SA, CLS, PRED) most strongly drive the readability-correctness decoupling.

4. **Practical guidelines** for integrating DRI scoring into code review workflows and CI/CD pipelines as a dual-quality gate alongside functional testing.

The remainder of this paper is structured as follows. Section II reviews related work. Section III presents the motivation and the DRI formulation. Section IV describes the study design. Section V presents results. Section VI discusses implications and limitations. Section VII concludes.

## II. Background and Related Work

### A. Code Readability: Measurement and Metrics

Code readability — the ease with which a developer can comprehend the intent and behavior of source code — has been a subject of empirical study for more than two decades. Buse and Weimer [8] introduced the first machine-learning model of code readability, using structural formatting features (line length, indentation, identifier density) to predict human readability judgments. Scalabrino et al. [9] extended this work by demonstrating that textual features — particularly the semantic quality of identifier names and the coherence between comments and code — are stronger predictors of readability than structural features alone. Subsequent work by Arnaoudova et al. [10] catalogued *linguistic antipatterns* — systematic naming defects that mislead developers about code behavior — and showed that such patterns correlate with higher defect density.

These studies establish that identifier quality is a primary driver of code comprehension. Lawrie et al. [11] showed empirically that full-word identifier names lead to significantly higher developer comprehension than abbreviated or single-letter names. Hofmeister et al. [12] found that descriptive variable names reduced bug-finding time by 19%. Allamanis et al. [13] demonstrated that neural models can learn naming conventions from large code corpora, achieving 94% top-5 accuracy on name suggestion tasks. Taken together, this body of work identifies identifier semantics — not just structural formatting — as the primary cognitive channel through which code communicates meaning to human readers.

### B. The IRAF-XADL Framework

The IRAF-XADL framework [7], published in the Engineering, Technology & Applied Science Research journal (Vol. 16, No. 3, 2026), is the measurement instrument used in this study. It provides identifier-level readability assessment using ten cognitive parameters extracted from source code via Abstract Syntax Tree (AST) analysis, combined with 768-dimensional CodeBERT embeddings and classified by a Self-Attention BiLSTM architecture.

The ten parameters are: **Meaningful Clarity (MC)**, the fraction of identifier tokens that are valid English words; **Naming Conformance (NC)**, adherence to language-specific naming conventions (PascalCase for classes, snake_case for functions and variables); **Optimal Length (OL)**, a Gaussian-weighted score penalizing excessively short or long identifiers; **Domain Relevance (DR)**, alignment of identifier tokens with the inferred domain vocabulary of the surrounding code; **Pronounceability (PR)**, a phonetic fluency score based on vowel-consonant ratios; **Lexical Familiarity (LF)**, average corpus frequency of identifier tokens from a reference vocabulary; **Context Consistency (CC)**, Jaccard similarity between an identifier's tokens and those of its peers in the same scope; **Scope Appropriateness (SA)**, a score relating identifier length to the size of its enclosing scope; **Cognitive Load Score (CLS)**, a composite of MC, LF, PR, and an ambiguity penalty; and **Predictability (PRED)**, the degree to which identifier tokens co-occur with neighboring identifiers in the codebase.

IRAF-XADL achieves 98.13% classification accuracy on Python and 98.42% on C++ code from the Kaggle Code Snippets: Insights & Readability dataset, producing a three-class output (High / Medium / Low readability) with per-class probability scores. A complementary snippet-level framework, ECRVR-MVEL [14], achieves comparable accuracy using a weighted ensemble of Graph Convolutional Networks, Deep Belief Networks, and Bidirectional Temporal Convolutional Networks, with LIME-based explanations confirming that the same cognitive parameters (MC and PRED in particular) drive snippet-level readability judgments.

The sophistication of this measurement instrument — cognitive semantic parameters plus deep language model embeddings at 98%+ accuracy — distinguishes this study from prior work on LLM code readability, which has relied on surface-level structural metrics [3] with substantially lower classification accuracy (77.5% in [3]).

### C. Readability of LLM-Generated Code

Ye et al. [3] conducted the most comprehensive study of readability in LLM-generated code to date, analyzing 5,869 matched code pairs (LLM-generated vs. human-written) using a 61-feature readability model synthesizing textual (TF), structural (BWF), information-theoretic (PF), and visual (DF) dimensions. Their study found that LLM-generated code achieves statistically higher average readability scores than human-written code (p < 0.001, r = 0.398), while exhibiting distinct issue patterns: LLM-generated code suffers disproportionately from Excessive Complexity and Redundant Comments, as well as three novel LLM-specific patterns (Unknown API usage, Redundant Variables, and Overblanking). Critically, Ye et al. use neither correctness labels nor functional test execution — their study measures only the readability dimension, leaving the relationship between readability and correctness unmeasured. Their Discussion section explicitly identifies this as a future research direction, describing a "novel critical correctness-readability gap" [3].

Takerngsaksiri et al. [15] surveyed 118 practitioners at Atlassian and found that 81% consider code readability important for AI-generated code, with 72% open to adopting LLMs specifically to improve readability. GPT-4-generated code was found to achieve readability scores comparable to human-written code across eight readability metrics, with negligible effect sizes. Horikawa et al. [16] examined commits made by AI agents in open-source repositories and found that while agents target readability in 28.1% of refactoring commits, the Maintainability Index actually decreased in 56.1% of these commits and Cyclomatic Complexity increased in 42.7%, suggesting a systematic divergence between intended and achieved readability improvements.

### D. Semantic Bugs and Silent Failures in LLM-Generated Code

Gao et al. [4] conducted a systematic literature review of 72 studies on bugs in LLM-generated code, constructing a comprehensive taxonomy spanning Functional, Syntax, Reliability, Code Style, Hallucination, and System categories. Their analysis shows that semantic errors — faults in which code appears syntactically correct but implements incorrect logic — constitute more than 60% of observed failures in several studied models, including DeepSeek-Coder-6.7B and QwenCoder-7B. This finding is corroborated by industry reports: Endor Labs (2025) found that 62% of AI-generated code contains security weaknesses or design flaws that are not caught by functional tests. The relationship between this high rate of semantic failure and the simultaneous high readability of LLM code [3] defines the central problem studied in this paper.

### E. Multi-Dimensional Code Quality Assessment

Wang et al. [17] investigated coding style inconsistencies in LLM-generated code using a taxonomy of 24 inconsistency types across five dimensions (Formatting, Semantic, Expression/Statement, Control Flow, and Fault Tolerance). Their study filtered to functionally correct code only before readability analysis, explicitly excluding incorrect code from consideration. This methodological choice means that their work measures the stylistic quality of *already-correct* code and cannot speak to the relationship between readability and correctness.

The RACE benchmark [18] evaluates LLM-generated code across four dimensions — Readability, mAintainability, Correctness, and Efficiency — as independent metrics. While it evaluates all four, it does not compute correlations between them or introduce a composite risk metric. He et al. [19] measured the effect of Cursor AI adoption in open-source projects and found a 25% increase in code complexity and 30–41% growth in technical debt within 90 days, attributing this to LLMs that "prioritize passing tests over non-functional requirements such as human readability" — a finding that further motivates the study of correctness-readability independence.

### F. Research Gap

The reviewed literature establishes two independent facts: (1) LLM-generated code is at least as readable as human-written code [3], [15]; and (2) LLM-generated code frequently contains semantic errors that appear syntactically correct [4]. No study has measured whether these two facts interact — specifically, whether the readability of LLM-generated code is correlated with, or independent of, its functional correctness. The Deceptive Readability Index proposed in this paper provides the first quantification of this interaction.

## III. Motivation and Problem Formulation

### A. Illustrative Examples

The following examples illustrate the failure mode we study. Each is a plausible LLM-generated Python function with high expected readability scores under IRAF-XADL — well-chosen names, conformant conventions, appropriate length — yet containing a semantic error.

**Example 1: Binary Search (Off-by-One)**

```python
def binary_search(sorted_list: list, target: int) -> int:
    """Find the index of target in sorted_list, return -1 if not found."""
    left_boundary = 0
    right_boundary = len(sorted_list)        # BUG: should be len(sorted_list) - 1
    while left_boundary <= right_boundary:
        middle_index = (left_boundary + right_boundary) // 2
        if sorted_list[middle_index] == target:
            return middle_index
        elif sorted_list[middle_index] < target:
            left_boundary = middle_index + 1
        else:
            right_boundary = middle_index - 1
    return -1
```

IRAF-XADL assessment: All five identifiers (sorted_list, target, left_boundary, right_boundary, middle_index) are full English words (MC = 1.0), follow snake_case convention (NC = 1.0), are appropriately descriptive for their scope (SA ≈ 0.85), and are highly familiar terms (LF ≈ 0.90). Overall readability class: **High**. Yet `right_boundary = len(sorted_list)` causes an IndexError on the first probe when the list is non-empty. This sample has DRI ≈ 0.91.

**Example 2: Median of a List (Wrong for Even-Length Input)**

```python
def calculate_median(number_list: list[float]) -> float:
    """Return the median value from an unsorted list of numbers."""
    sorted_numbers = sorted(number_list)
    middle_position = len(sorted_numbers) // 2
    return sorted_numbers[middle_position]   # BUG: returns upper middle for even-length
```

IRAF-XADL assessment: number_list, sorted_numbers, middle_position — all highly meaningful, conformant, and familiar. Readability class: **High**. The function silently returns the wrong value for even-length inputs (e.g., [1, 3, 5, 7] returns 5 instead of 4.0). DRI ≈ 0.74 (passes approximately 26% of test cases in a rigorous suite).

**Example 3: Word Frequency Counter (Hallucinated Method)**

```python
def count_word_frequencies(text_content: str) -> dict[str, int]:
    """Count how many times each word appears in the provided text content."""
    normalized_text = text_content.lower_and_strip()    # BUG: method does not exist
    word_tokens = normalized_text.split_on_whitespace() # BUG: method does not exist
    frequency_mapping = {}
    for individual_word in word_tokens:
        frequency_mapping[individual_word] = frequency_mapping.get(individual_word, 0) + 1
    return frequency_mapping
```

IRAF-XADL assessment: Identifiers (text_content, normalized_text, word_tokens, frequency_mapping, individual_word) are among the highest-quality names possible for this task. Readability: **High**. Yet `lower_and_strip()` and `split_on_whitespace()` do not exist in the Python standard library. The code fails immediately with an AttributeError. DRI = 1.0.

These three examples share a common pattern: the naming quality is genuinely excellent by every cognitive parameter, yet the code is entirely or substantially non-functional. This is the readability-correctness decoupling.

### B. The Deceptive Readability Index

We formalize the deceptive readability phenomenon through the following metric.

**Definition 1 (Deceptive Readability Index)**. For a code sample *c*, let *P_High(c)* denote the probability of the "High" readability class assigned by IRAF-XADL, and let *pass_ratio(c)* ∈ [0, 1] denote the fraction of ground-truth test cases that *c* passes. The Deceptive Readability Index is:

```
DRI(c) = P_High(c) × (1 − pass_ratio(c))
```

**Interpretation**: DRI(c) = 0 when the code is either unreadable (P_High ≈ 0, flagged by readability tools) or fully correct (pass_ratio = 1, verified by tests). DRI(c) reaches its maximum (≈ 1.0) when code is confidently classified as highly readable but fails all test cases — the case of maximum deceptive risk to a human reviewer.

We further define risk tiers:

| DRI Range | Interpretation | Review Priority |
|-----------|---------------|-----------------|
| 0 | Unreadable or fully correct | Low |
| (0, 0.3) | Low deception risk | Normal |
| [0.3, 0.6) | Moderate deception risk | Elevated |
| [0.6, 1.0] | High deception risk — readable but substantially wrong | Critical |

For comparative analysis across LLMs:

```
DRI@Model = (1/N) × Σ DRI(cᵢ)  for i = 1 to N
```

A model with high DRI@Model systematically generates readable-looking but incorrect code — posing the highest aggregate risk in an AI-assisted development context.

### C. Research Questions

Motivated by the foregoing analysis, this study addresses four research questions:

**RQ1**: Is there a statistically significant difference in IRAF-XADL readability scores between functionally correct and incorrect LLM-generated code?

**RQ2**: Does the IRAF-XADL readability score predict the functional correctness of LLM-generated code?

**RQ3**: Does the magnitude of the readability-correctness gap, as measured by DRI@Model, vary across LLM architectures?

**RQ4**: Which of the ten IRAF-XADL cognitive parameters (MC, NC, OL, DR, PR, LF, CC, SA, CLS, PRED) contribute most to deceptive readability in LLM-generated code?

## IV. Study Design

### A. Overview

We conduct an observational empirical study applying the IRAF-XADL readability assessment framework to a large corpus of LLM-generated Python code with ground-truth correctness labels. The study pipeline proceeds in four stages: (1) acquisition of LLM-generated code and correctness labels from established benchmarks; (2) readability scoring of each sample using IRAF-XADL; (3) DRI computation; and (4) statistical analysis addressing each research question. The full replication package, including datasets, scoring scripts, and analysis code, will be made publicly available upon acceptance.

### B. Benchmarks

We use two established functional programming benchmarks, both available through the EvalPlus framework [20], which provides rigorous extended test suites that substantially reduce false positives present in the original benchmarks.

**HumanEval+**: The HumanEval benchmark [6] comprises 164 handwritten Python programming problems, each with a natural-language docstring, a function signature, and a set of test cases. The EvalPlus extension (HumanEval+) augments each problem with approximately 80× more test cases than the original, ensuring that a passing solution is genuinely correct rather than simply passing a small number of examples. HumanEval+ is the primary benchmark for this study.

**MBPP+**: The Mostly Basic Python Problems benchmark [21] comprises 378 Python programming problems (after removal of syntactically broken tasks by EvalPlus) with ground-truth test cases amplified by approximately 35×. MBPP+ is used as the secondary validation benchmark.

Together, the two benchmarks provide 542 distinct programming problems covering a broad range of algorithmic and data manipulation tasks in Python.

### C. LLM Solution Acquisition

We use pre-generated solution files from the EvalPlus v0.1.0 release [20], which provides complete generated solution sets for a range of models in JSONL format, along with per-sample evaluation results. For each model, we use the greedy (temperature = 0) or the closest available temperature setting, selecting one solution per problem to ensure comparability across models.

**Selected models** (spanning a capability range for RQ3 analysis):

| Model | Parameters | Category |
|-------|------------|----------|
| CodeLlama-7B-Python | 7B | Smaller open-source |
| CodeLlama-34B-Python | 34B | Larger open-source |
| WizardCoder-34B | 34B | Instruction-tuned open-source |
| DeepSeek-Coder-33B | 33B | Recent open-source frontier |
| StarCoder2-15B | 15B | Permissive-license open-source |

This selection spans a capability range and provides sufficient diversity for the RQ3 inter-model comparison. All models are available under open licenses, and their pre-generated EvalPlus solutions are publicly downloadable.

**Total dataset**: 542 problems × 5 models = 2,710 code samples.

### D. Correctness Labeling

For each (model, problem) pair, we execute the generated solution against the full EvalPlus test suite using the `evalplus.evaluate` command-line tool. Each execution is run in a sandboxed Docker container with a 10-second timeout per test case to prevent infinite loops. We record:

- `pass_ratio(c)`: the fraction of test cases passed, ∈ [0, 1]
- `correct(c)`: a binary label, 1 if pass_ratio = 1.0, 0 otherwise

This provides the ground-truth correctness dimension for DRI computation and for all statistical analyses.

### E. Readability Scoring

Each code sample is scored using the IRAF-XADL prediction API [7]. The API accepts a Python code string and returns:

- **P_High, P_Medium, P_Low**: class probabilities from the SA-BiLSTM classifier
- **10 feature values**: per-identifier averages for MC, NC, OL, DR, PR, LF, CC, SA, CLS, PRED
- **Structural metrics**: lines of code (LOC), cyclomatic complexity, number of identifiers, indentation depth, loop count

We use a batched REST API call (`/batch` endpoint) to score all 2,710 samples in a single pipeline run against the locally-hosted IRAF-XADL service. The model checkpoint used is the production artifact (`artifacts/iraf_xadl_augmented.pt`) trained on the full Kaggle Code Snippets: Insights & Readability dataset as reported in [7].

### F. DRI Computation

For each sample, DRI is computed as:

```
DRI(c) = P_High(c) × (1 − pass_ratio(c))
```

We additionally compute DRI@Model for each of the five models and analyze the DRI distribution across correctness groups (correct vs. incorrect) and across models.

### G. Statistical Analysis Plan

**RQ1 — Readability distribution by correctness**:
We partition all samples into two groups: Correct (pass_ratio = 1.0) and Incorrect (pass_ratio < 1.0). We compare the distribution of P_High between groups using the two-sided Mann-Whitney U test [22] (non-parametric, appropriate for non-normal distributions). We report the U statistic, p-value, and Cohen's d effect size. We additionally report the proportion of Incorrect samples that receive a "High" readability classification — the empirical false-signal rate.

**RQ2 — Readability as a predictor of correctness**:
We fit a logistic regression model predicting `correct(c)` from P_High(c) and report the Area Under the ROC Curve (AUC). We additionally compute the Spearman rank correlation ρ between P_High(c) and pass_ratio(c) across all samples. An AUC below 0.65 is interpreted as confirming that readability score is a poor predictor of correctness.

**RQ3 — Inter-model DRI comparison**:
We compare DRI@Model across the five LLMs using the Kruskal-Wallis H test, followed by pairwise Dunn's post-hoc test with Bonferroni correction. We report DRI@Model and the proportion of High-DRI samples (DRI ≥ 0.6) per model.

**RQ4 — Parameter-level attribution**:
We fit a logistic regression model predicting `incorrect and readable` (DRI ≥ 0.3) from the ten IRAF-XADL parameter values as independent variables. We report coefficient magnitudes (normalized) as feature importance scores, indicating which parameters most strongly distinguish deceptively readable code from other code. We additionally compute per-parameter Spearman correlations with DRI.

**Benjamini-Hochberg correction** is applied to all multiple comparisons to control the false discovery rate at α = 0.05.

### H. Threats to Validity

**Internal validity**: The IRAF-XADL model was trained on the Kaggle Code Snippets dataset, which comprises human-written code. Its calibration may not perfectly transfer to LLM-generated code with different distributional properties (e.g., lower incidence of abbreviations, higher use of descriptive names). We mitigate this by analyzing the per-parameter breakdown (RQ4) separately, which does not depend on classifier calibration.

**Construct validity**: DRI uses a binary correctness signal (pass_ratio derived from test cases). Test suites may not cover all edge cases, meaning some "correct" samples may contain latent bugs not caught by the test suite. EvalPlus mitigates this substantially through its 35–80× test case amplification.

**External validity**: Our study uses Python code from algorithmic benchmark problems (HumanEval+, MBPP+). Results may not generalize to other programming languages, to application-level code with complex dependencies, or to other LLM families not included in the study.

**Conclusion validity**: Statistical tests are chosen to be appropriate for non-normal distributions. Effect sizes are reported alongside p-values to distinguish statistical from practical significance.

## VI. Discussion

### A. Implications for Developers

If our hypothesis holds — that incorrect LLM code shows no lower readability than correct code — then readability review alone provides no meaningful signal about functional quality in AI-assisted development. Developers should not treat well-named, well-structured LLM-generated code as a positive quality signal. The verification paradox documented by Stack Overflow 2025 [5] — 96% stated distrust, 48% actual verification — may be partly explained by the false reassurance that readable-looking code provides.

**Practical recommendation**: DRI scoring can serve as a lightweight first-pass risk filter in code review workflows. A commit containing one or more High-DRI functions (DRI ≥ 0.6) should be flagged for mandatory test execution before merge — regardless of code review outcome.

### B. Implications for CI/CD Integration

The DRI can be integrated into existing CI/CD pipelines as follows:

1. Static readability score: computed by IRAF-XADL on the submitted code
2. Functional correctness: from the existing test suite pass rate
3. DRI computation: P_High × (1 − pass_ratio)
4. Gate: flag any function with DRI ≥ 0.6 for human review escalation

This creates a *dual-quality gate* that existing correctness-only gates miss: a function that passes all tests has DRI = 0 (no flag); a function that fails some tests but looks unreadable has low P_High (low DRI, developer already skeptical); the dangerous case — readable and failing — is caught by the DRI gate.

### C. Limitations

The study is limited to Python code from algorithmic benchmarks. DRI behavior on application-level code, on stateful systems, or on code in other languages (Java, C++, TypeScript) is unknown. Future work should validate DRI across these dimensions. Additionally, our readability scorer is trained on human-written code; fine-tuning IRAF-XADL on LLM-generated code may improve calibration and is a direction for future work.

### D. Relation to Prior Work

Our findings complement rather than contradict Ye et al. [3]. Their study establishes that LLM code is at least as readable as human code — a finding we use as the premise for our study. Where they measure readability as an end in itself, we measure its relationship to correctness as a means to assess developer trust calibration. The "novel critical correctness-readability gap" they identify [3] is precisely the gap we quantify.

## VII. Conclusion and Future Work

This paper has presented the first empirical measurement of the relationship between cognitive readability scores and functional correctness in LLM-generated code. We introduced the Deceptive Readability Index (DRI = P_High × (1 − pass_ratio)) as a formal characterization of the false-trust risk that readable-but-incorrect code poses to human reviewers. Using the IRAF-XADL cognitive parameter framework — a 98%+-accurate identifier-level readability assessment system — applied to [N] LLM-generated Python solutions from HumanEval+ and MBPP+, we demonstrated that [key findings pending experimental results].

The core implication is that readability, as a quality signal for LLM-generated code, requires a correctness correction. Readable code from an LLM is not more likely to be correct than unreadable code. The verification paradox documented in developer surveys — widespread stated distrust combined with widespread actual non-verification — is not irrational: it may reflect a real, if unconscious, reliance on readability as a surrogate for correctness. DRI scoring can help developers calibrate this trust more accurately, and its integration into CI/CD pipelines as a dual-quality gate is a concrete, actionable recommendation.

Future work will extend this analysis in three directions. First, we will evaluate DRI behavior on application-level code from real-world repositories, where the semantic complexity is higher and the test coverage is typically lower than in benchmark settings. Second, we will extend the study to multi-agent code generation pipelines, where one LLM writes code and another reviews it, to investigate whether AI reviewers are susceptible to deceptive readability in the same way human reviewers are. Third, we will fine-tune the IRAF-XADL readability scorer on LLM-generated code to improve parameter calibration for this distribution.
